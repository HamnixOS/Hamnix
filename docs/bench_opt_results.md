# Adder optimizer performance scorecard — 5-way (AST-only vs full `--opt`)

Auto-generated by `bash scripts/bench_opt.sh` (`scripts/_bench_opt_run.py`). Re-run to refresh.

> **Why the AST-only lane.** The full Adder `--opt` (~1.83× gcc-O2) arms ten high-level AST passes **plus** six codegen levers (register allocator, IR-emit, strength-reduce, instruction-select, vectorizer, cmp/jcc fusion). The register allocator has a kernel-target miscompile that blanks the desktop, so full `--opt` cannot ship yet. The **AST-only** config keeps the ten AST passes ON and turns the six codegen levers OFF via `ADDER_OPT_DISABLE=regalloc,iremit,strengthreduce,isel,vec,cmpjcc` — this config is correct on the kernel and **shippable today**. This scorecard quantifies how much of the full-opt win AST-only banks now vs what the register-allocator fix would add.

## What is compared

Five configurations on identical compute kernels (`tests/bench/opt/<name>.{ad,c}`):

1. **Adder-O0** — native `codegen.ad`, no `--opt` (baseline stack-machine backend).
2. **Adder AST-only** — `--opt` with the six codegen levers disabled (`ADDER_OPT_DISABLE=regalloc,iremit,strengthreduce,isel,vec,cmpjcc`): only the ten AST passes run. **Shippable today** — unaffected by the kernel-target regalloc bug.
3. **Adder full-ON** — full `--opt` / `ADDER_OPT=1`: all ten AST passes AND the six codegen levers (blocked on the register-allocator fix for the kernel target).
4. **C, gcc -O0** — unoptimized C.
5. **C, gcc -O2** — optimized C.

## Methodology — why it is fair

The native Adder compiler's `x86_64-linux` host path (`tests/fuzz/ad_codegen_host.py`) runs lex → parse → opt → `codegen.ad` and **wraps the emitted machine code into a real `ELF64` `EM_X86_64` executable** that runs natively on this host CPU — the *same CPU* and *same ELF ABI* as the gcc-compiled C binaries. All five configs are therefore timed as ordinary host processes on one CPU: no QEMU, no Hamnix image, no cross-VM timer skew. The only difference among the three Adder lanes is which optimizer stages ran; between Adder and C, only the compiler. (The C binaries are timed on the host rather than inside an `enter linux {}` namespace for exactly this reason — keeping every config on one bare CPU is the fairest control.)

The **AST-only** lane is produced by setting `ADDER_OPT_DISABLE=regalloc,iremit,strengthreduce,isel,vec,cmpjcc` in the environment for that compile; the codegen driver reads it and skips the six sibling codegen levers, so only the ten high-level AST passes fire. The full-ON lane sets nothing (all levers armed by `--opt`).

**Correctness before speed.** Each kernel prints one decimal checksum; the harness asserts it is *identical* across the builds before timing. The Adder-O0 and both C builds must agree (that pins the reference answer); a kernel whose full-ON *or* AST-only build disagrees is reported as a **miscompile finding** and excluded from that lane. AST-only is expected to agree everywhere (the regalloc bug is kernel-target-specific, not triggered by these userspace kernels); a disagreement would itself be a finding.

**Noise control.** Each binary is warmed up once (discarded), then timed **best-of-7** (minimum wall time). Iteration counts are sized so every kernel runs well over 100 ms even at C -O2, dwarfing process-spawn + timer overhead. The kernels that are pure ALU/closed-form (`licm`, `dcecopy`) write into a global ring buffer each step so gcc -O2 cannot delete the loop — the comparison stays about generated loop code, not gcc's loop-elimination.

## Headline

- **AST-only captures 4% of the full `--opt` speedup.** AST-only is **1.14× over O0**; full `--opt` is **4.20× over O0** (geomean over the correct kernels). The remaining ~96% is what the six codegen levers (led by the register allocator) would add once the kernel-target regalloc bug is fixed.
- **vs gcc -O2** (the `× C-O2` slower-than ratio; lower is better, <1 = faster than gcc): full `--opt` holds **1.41× gcc-O2** — within the project's ≤2× target (historically ~1.83×) — while AST-only alone is **5.21× gcc-O2**, i.e. it leaves most of the C-competitive speed on the table until the codegen levers land.
- **One-liner:** AST-only captures **4%** of the full `--opt` speedup; AST-only is **5.21× gcc-O2** vs full's **1.41× gcc-O2**.

## AST-only correctness

Every timed kernel's AST-only build produces a checksum **identical** to the Adder-O0 baseline — the ten AST passes are correct on this suite, confirming the config is shippable today.
