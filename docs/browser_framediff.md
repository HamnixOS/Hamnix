# Browser fidelity harness — pixel scorecard (framediff-gfx)

USER directive: **hambrowse must "render like Firefox and Chrome."** This is the
measurement + gap ranking that drives it.

The harness renders every corpus page three ways on the dev host (no QEMU):

- **hambrowse (pixel)** — `user/hambrowse_host_gfx.ad` → `lib/htmlpage` +
  `lib/htmlpaint`, the *same* layout + paint code that runs on device;
- **chromium** — headless `--screenshot`, matched viewport, opaque white bg;
- **firefox** — headless `--screenshot`, matched viewport, opaque white bg.

All three are driven with the **same DejaVu faces** (`fonts/dejavu-*.ttf`, also
installed system-wide) at the same default UA sizes, so the corpus pages use
`font-family:"DejaVu Sans"`. That makes the RMSE track **layout + paint**
(box geometry, borders, radii, shadows, baselines, spacing) rather than a font
substitution. Geometry is normalized by `scripts/framediff_gfx_prep.py`
(flatten-white → content-bbox trim → resize reference to the hambrowse content
box) and scored with ImageMagick `compare` (RMSE + a `-fuzz`'d absolute-error
count that tolerates anti-alias noise).

**Lower RMSE == closer to the reference.** chromium and firefox should broadly
**agree** per page — where they do, that cross-validates the number; where they
diverge, the reference itself is ambiguous. This supersedes the old text-mode
harness (`scripts/framediff_run.sh`), which diffed hambrowse's monospace host
dump and so misrepresented proportional text
(`feedback_host_preview_monospace_lies`).

## Ranked pixel-level gaps

_Hand-authored from the heatmaps/side-by-sides above; ranked by visual impact so
the engine team closes the biggest Chrome/Firefox gaps first. Regenerate the
images with `scripts/framediff_gfx_all.sh`, then re-rank._

Ranked worst-impact first. Each is an actionable engine target with the page(s)
whose heatmap shows it. Cross-validated: chromium and firefox agree within <1%
normalized RMSE on every page, so these are real hambrowse gaps, not reference
noise.

1. **`flex-grow` / `flex:1` spacer collapses — flex items are not justified to
   the far edge.** `.spacer{flex:1}` takes zero width, so nav items stay bunched
   at the left instead of being pushed to the right edge.
   _Pages: hero (0.50, #1), flexnav (0.43, #2) — the two worst pages._
   Target: implement free-space distribution to `flex-grow` children (and by
   extension `justify-content`).

2. **Row flex container does not fill the line box; flex/grid items do not
   stretch to the cross size (`align-items:stretch`).** The nav bar and hero
   `background` paint only ~half width (container shrinks to content instead of
   100%), and the docs sidebar `background:#f4f6fb` covers only its text height,
   not the full column. Truncated background blocks are a very visible gap.
   _Pages: hero, flexnav, docs (0.21)._
   Target: default a block-level flex container to 100% inline size; default
   `align-items:stretch` so a flex/grid item's cross size fills the row/column.

3. **Table borders and cell backgrounds are not painted.** `border:1px solid`
   on `th`/`td`, `border-collapse`, `th{background}`, and `tr:nth-child(even)`
   zebra are all absent — hambrowse draws bare text columns where Chrome/Firefox
   draw a full ruled, shaded grid. High impact for any tabular data.
   _Page: table (0.29, #4)._
   Target: paint collapsed cell borders + cell/row background fills; wire
   `:nth-child()` to the background.

4. **Horizontal margins on inline elements are dropped.** `.nav a{margin-left:16px}`
   collapses, so links render glued together ("HomeDocsAbout").
   _Pages: hero, flexnav._
   Target: apply left/right margin to inline boxes when advancing the pen.

5. **Form-control chrome is missing/thin.** `<input>` renders as a thin
   underline rather than a bordered, padded box; `<select>` has no border or
   dropdown affordance; control heights are too short, so the vertical rhythm of
   the whole form is compressed vs the reference.
   _Page: form (0.23)._
   Target: draw the UA input/select/button box (2px inset border, padding,
   `size`-driven width, min-height).

6. **Grid item text lands in the first track; cell height collapses to one
   line.** The three cell *backgrounds* are placed in 3 columns, but every
   item's text is painted at the first cell's x ("AlphaBravoCharlie"
   concatenated), and cells are one line tall instead of padded.
   _Page: grid (0.14)._
   Target: place each grid item's inline content at its own cell origin; honor
   cell padding/min-content height.

7. **`border-radius` pill not applied to the inline-block badge; badge
   mis-positioned.** Card *block* corners round correctly, but `.badge{border-radius:12px}`
   renders as a square box, and the badge sits low, overlapping the card's
   bottom edge, instead of inline after the text with `margin-top`.
   _Page: cards (0.22)._
   Target: apply large/`>=half-height` radii to inline-block backgrounds; fix
   inline-block vertical placement + top margin.

8. **Line-height / leading is too tight — pages render ~10–15% shorter.** Body
   paragraphs pack closer than the reference; `line-height:1.5` is
   under-applied, compressing the whole page vertically (visible as diff
   ghosting after the normalizer resizes the taller reference onto hambrowse's
   shorter box).
   _Pages: text (0.23), docs, article._
   Target: use the CSS `line-height` (unitless × font-size) for the line-box
   height, not a fixed leading.

9. **Block top margins (headings) are too small.** `h1`/`h2` sit higher with
   less space above than the UA/author margins call for.
   _Pages: text, docs._
   Target: honor `margin-top` on block boxes (and UA heading margins); check for
   margin-collapsing under/over-application.

10. **Explicit `height` on a floated box is not honored.** `.hero{height:80px}`
    renders visibly taller than 80px.
    _Page: article (0.29, #3)._
    Target: clamp a floated/block box to its specified `height`.

11. **Inline-block button background/padding not painted.** The hero
    `.cta{background:#fff;border-radius;padding}` renders as bare blue text with
    no white rounded button box.
    _Page: hero._
    Target: paint background + border-radius + padding box for inline-block.

12. **`font-family:monospace` is ignored (kept out of the layout metric).** The
    engine paints proportional DejaVu Sans regardless of a `monospace` family,
    so code/terminal-style pages mis-measure. The corpus uses `DejaVu Sans` so
    this one bug doesn't drown every page's RMSE, but it is a real fidelity gap.
    Target: select the mono TTF face when the computed `font-family` resolves to
    `monospace`.

## Reading the score

- The metric is **relative**, not pixel-parity. A residual floor comes from
  hinting/anti-alias differences between hambrowse's rasterizer and the
  reference engines even when layout is identical; the `-fuzz`'d AE count is the
  better "how many pixels are *really* wrong" signal.
- The corpus deliberately uses `DejaVu Sans` because that is the face the pixel
  engine paints. hambrowse currently **ignores `font-family:monospace`** and
  paints proportional sans regardless — a real gap, tracked in the ranked list,
  but kept *out* of the layout metric so one font bug doesn't drown every page.
- Pages are self-contained and deterministic (inline CSS, no network), so the
  numbers are reproducible run-to-run.
