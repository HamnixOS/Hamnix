# Native Browser (hambrowse) — W3C / ECMAScript Conformance Roadmap

From a 14-agent parallel audit (2026-07-16). Dual-target, host-iterated (~40 gates). ~30-50% conformant/area; full conformance needs real spec ALGORITHMS per subsystem — enabled by the modular lib/web/ tree refactor.

## ECMAScript rounds landed (lib/web/js/*, host gates in scripts/test_jsengine_*_host.sh)

Deferred: apply/construct/defineProperty/getOwnPropertyDescriptor/get(set)PrototypeOf traps, revocable. `test_jsengine_proxy_host.sh`.
  - **Typed arrays + ArrayBuffer** — byte-backed (shared pool, little-endian pointer load/store): Int8/Uint8/Uint8Clamped/Int16/Uint16/Int32/Uint32/Float32/Float64. Construct from length / array / iterable / ArrayBuffer (shared view); indexed get/set with ToInt32+mask (clamp for Uint8Clamped); length/byteLength/byteOffset/buffer/BYTES_PER_ELEMENT; set()/subarray()/slice()/fill(); for-of/spread/Array.from; static of()/from(). - **R8** (numeric literals + JSON reviver):
  - **Numeric literal forms** — ES2021 numeric separators `_` in decimal/hex/binary/octal integers + fraction + exponent (`1_000_000`, `0xFF_FF`, `1e1_0`); binary `0b`/`0B` and octal `0o`/`0O` integer literals; BigInt `n` suffix (originally kept as a float64 Number — **now a true arbitrary-precision BigInt as of R11**, see below). Fixed in `lib/web/js/lexer.ad` (lex_number) + `lib/web/js/value.ad` (parse_num_bytes). Previously the lexer split `1_000` into `1` + identifier `_000` (ReferenceError) and read `0b101` as `0` + identifier — a hard abort of any modern minified/transpiled bundle using these forms. `test_jsengine_numliteral_host.sh`.
  - **JSON.parse(text, reviver)** — the optional second argument is now honored: a bottom-up InternalizeJSONProperty walk calls `reviver.call(holder, key, value)` at every node (children first), deletes the property on an `undefined` result, handles arrays by index, threads the holder object as `this`, and invokes the root with key `""`. Previously the reviver was silently ignored (parsed data returned untransformed — a correctness bug). Fixed in `lib/web/js/builtins/collections.ad` (json_parse_val + json_internalize/json_intern_props/json_reviver_call). `test_jsengine_numliteral_host.sh`. (JSON.stringify's `replacer` argument was subsequently landed in R10 — see above.)
- **R9** (class private members + static blocks + global symbol registry):
  - **Class private `#fields` / `#methods` / accessors** — the lexer now tokenizes `#name` as `TK_PRIVATE`, retaining the leading `#` so a private slot's key can never collide with a normal string property. Private instance fields (`#x = v`), private methods (`#m(){}`), private getters/setters (`get #x(){}`), and private static members (`static #n = v`) all parse and evaluate; `this.#x` / `Ctor.#n` access works through the ordinary member get/set path (stored under the `#`-prefixed key). The `#x in obj` **brand check** is a dedicated `ND_PRIVNAME` LHS handled by the `in` binop (never evaluated as a value; probes the private slot through the prototype chain). Crucially, private slots are HIDDEN: `Object.keys`, `for-in`, and `JSON.stringify` skip any key beginning with `#` (via a new `is_hidden_key` in `lib/web/js/value.ad`). Fixed across `lib/web/js/{lexer,parser,interp,value}.ad` + `builtins/collections.ad`. `test_jsengine_es2022priv_host.sh`.
  - **`Symbol.for(key)` / `Symbol.keyFor(sym)`** — a process-wide symbol registry interns by string key so `Symbol.for("a") === Symbol.for("a")` (and stays distinct from a plain `Symbol("a")`); the registration key is stored back on the symbol as a hidden `@@forkey` own property for `Symbol.keyFor` reverse lookup (returns `undefined` for an unregistered symbol; throws `TypeError` on a non-symbol). Added in `lib/web/js/builtins/r7.ad` (symbol_for/symbol_key_for) + registered in `setup.ad`. `test_jsengine_es2022priv_host.sh`.
  - **`static { }` initializer blocks** — the class-body parser now recognizes a bare `{ ... }` member after `static` (`ND_METHOD` with `n_c=3`); `eval_class` runs the block bodies interleaved with `static` fields in SOURCE ORDER, with `this` = the class object and the class's own name bound in a class-scope env (so `static { C.x = ... }`, `static y = C.z`, and `this.#priv` inside a block all resolve). Added in `lib/web/js/parser.ad` (parse_class) + `interp.ad` (eval_class static pass). `test_jsengine_es2022priv_host.sh`.
  - **Bonus fix (symbol-key leak)** — the same `is_hidden_key` filter also stops symbol-keyed properties (`@@sym:` keys) from leaking into `Object.keys` / `JSON.stringify`, a pre-existing correctness bug (`o[sym]=v` previously showed `@@sym:NNN` in enumeration output).
- **R10** (`for await...of` + JSON.stringify replacer):
  - **`for await (x of asyncIterable)`** async iteration — the parser recognizes a contextual `await` after `for` (inside an async body) and marks `ND_FOROF` (`n_d=1`); `exec_forof` dispatches to a new `exec_forawait` (`lib/web/js/interp.ad`). It first tries the source's `@@asyncIterator` (a fresh global `g_sym_async` symbol, also installed on the generator prototype so an `async function*` is its own async iterator): it calls `next()`, **awaits** the result via the engine's existing deterministic `do_await` (microtask-drain) to get `{value, done}`, and binds/executes the loop body per element (honoring `break`/`continue`/labels through the shared `forof_bind`/`forof_after`). If there is no `@@asyncIterator`, it falls back to the **async-from-sync wrapper**: it materializes the sync iterable via `collect_iterable_into` and **awaits each produced value** — so `for await (x of [Promise.resolve(1), 2, ...])` yields the resolved values in order. Because `await` drains the deterministic queue, the whole loop completes synchronously (no call-stack suspension), matching the engine's eager async/generator model. `test_jsengine_asynciter_host.sh`.
  - **`JSON.stringify(value, replacer[, space])`** — the 2nd argument is now honored. A **function replacer** `fn(key, value)` is called (with the holder as `this`) per property, its result substituted; an **array replacer** is an allowlist of object keys to include (applied at every object level, not to array indices). `value.toJSON(key)` is also honored on this path. Because a replacer function does its own string-building (which cannot run while the shared build cursor is held by `json_emit` — the same two-phase hazard as `String.replace`), the value tree is **pre-transformed** into a fresh tree (`json_transform`/`json_rep_value`/`json_key_allowed`, root holder `{"": value}`) BEFORE the emitter runs; the no-replacer path stays on the fast direct emitter (byte-identical to before). Added in `lib/web/js/builtins/collections.ad` (`json_stringify_full` + helpers). `test_jsengine_asynciter_host.sh`.
- **R11** (true arbitrary-precision BigInt — closes the largest ES gap):
  - **BigInt value type** — a new `TAG_BIGINT` value tag (`lib/web/js/consts.ad`) whose `v_ref` indexes a new immutable bignum record: a **sign + base-2^32 little-endian limb magnitude** (`lib/web/js/bigint.ad`, pools `bi_sign`/`bi_off`/`bi_len` over a shared `bi_limbs` uint32 bump pool). Records are immutable, so a sign flip shares limbs with no copy. `123n` / `0xffn` / `0b1010n` / `0o17n` literals now LEX to a distinct `TK_BIGINT` token carrying the source digits (lexer keeps the text sans `n`; the regex/divide heuristic treats a BigInt literal as a value so `7n / 2n` is division, not a regex) and PARSE to `ND_BIGINT`, evaluated by exact `bi_parse` — no more float64 precision loss past 2^53.
  - **Arithmetic** — exact `+ - * / % **` on two BigInts: schoolbook magnitude add/sub/mul, **bit-by-bit long division** (÷ **truncates toward zero**, `%` takes the dividend's sign), and exponentiation-by-squaring. `++`/`--` and compound `+= -= *= …` stay BigInt. Mixing a BigInt with a Number in an arithmetic/bitwise op throws **TypeError** (per spec); unary `+` on a BigInt throws; `/ 0n` and a negative `**` exponent throw **RangeError**.
  - **Comparisons** — `< > <= >=` and `== ===` including exact **BigInt-vs-Number numeric compare** (`2n < 3`, `5n == 5` true, `5n === 5` false — `===` is false across types); loose `==` also handles BigInt-vs-String/Boolean by mathematical value.
  - **Bitwise** — `& | ^` implemented with **infinite-width two's-complement** semantics (correct for negative operands via a to/from two's-complement conversion); `<<` is a magnitude shift keeping sign; `>>` is an **arithmetic (floor) shift** (`-8n >> 1n === -4n`, `-7n >> 1n === -4n`); `~x === -(x+1)`. `>>>` throws TypeError (BigInts have no unsigned shift).
  - **Conversions / methods** — `typeof 1n === "bigint"`; `String(1n)` / `` `${1n}` `` / `1n.toString(radix)` (2–36); truthiness (`0n` falsy); `BigInt(number|string|boolean|bigint)` (non-integer Number → RangeError); `BigInt.asIntN(bits,x)` / `BigInt.asUintN(bits,x)`. `JSON.stringify` on a BigInt (or an object containing one) throws **TypeError**, per spec. Console/display prints the digits with no trailing `n` (String() semantics). Touched: `lib/web/js/{bigint,consts,state,lexer,parser,interp,setup}.ad` + `builtins/{collections,strings}.ad`. `test_jsengine_bigint_host.sh` (40 asserts vs known results: `2n**100n`, `50!`, div/mod signs, two's-complement bitwise, mixing/JSON TypeErrors).
  - Coverage is honest and broad: bitwise AND/OR/XOR/shifts (incl. negatives), `asIntN`/`asUintN`, and all six arithmetic ops are DONE. NOT done: `BigInt64Array`/`BigUint64Array` typed arrays, `DataView` get/setBigInt64, and a `Number(bigint)` that round-trips huge magnitudes exactly (uses a float64 approximation for out-of-int64 comparison fallbacks only).
  - Next JS gap: async iterator `return()`/`throw()` propagation and `Symbol.asyncIterator` on user objects returning promises from `next()` (partially exercised; async generators covered); then `BigInt64Array`/`DataView` BigInt accessors.
- **R12** (destructuring ASSIGNMENT + `window`-shaped globals):
  - **Destructuring as an ASSIGNMENT statement** — `[a,b]=[b,a]` (swap), `[x,,y]=arr` (elision/skip), `[a,...rest]=arr` (rest), `({p,q}=obj)`, `({name:nm}=obj)` (rename), object rest `({a,...others}=obj)`, element/property **defaults** (`[m=5]`, `{v=5}`), nested patterns (`[p,[q,r]]=…`, `({user:{id,tags:[t0,t1]}}=…)`), and member/index leaf targets (`[o.x,a[0]]=…`) all work. Previously this did NOTHING: `assign_to` handled only ident/member/index LHS, so an array/object **literal** LHS was silently ignored — only the `let/const/var` DECLARATION path (`bind_pattern`) understood patterns. Root cause + fix: added `assign_array_pattern`/`assign_object_pattern` in `lib/web/js/interp.ad`, dispatched from `assign_to` on an `ND_ARRAY`/`ND_OBJECT` target; they walk the literal's element list and assign each leaf via `assign_to` (recursing for nested patterns). Because `eval_assign` already fully evaluates the RHS before calling `assign_to`, the swap is correct. Two supporting parser fixes (`lib/web/js/parser.ad`): array-literal **elisions are now preserved** as a new `ND_HOLE` node (`parse_array_lit`) so `[x,,y]=arr` maps `y←arr[2]` (a hole also now reads as `undefined` with correct `.length` in a value context, via `eval_array_lit`); and object shorthand **CoverInitializedName** `{x = default}` parses (wrapping the shorthand in an `ND_ASSIGN`) so `({v=5}=obj)` applies its default. Declaration destructuring is unchanged and re-asserted green. `test_jsengine_destructure_host.sh`.
  - **`window`-shaped globals** — the existing `window`/`globalThis`/`self` alias now also carries the browser viewport/URL data properties pages read ungated: `innerWidth`/`innerHeight`/`outerWidth`/`outerHeight` (default 1024×768), `scrollX`/`scrollY`/`pageXOffset`/`pageYOffset` (0), `devicePixelRatio`/`screenX`/`screenY`, a minimal `location` object (`href`/`protocol`/`host`/`hostname`/`port`/`pathname`/`search`/`hash`/`origin`), and a `screen` object. Defined on the global scope (like `navigator`) so both `window.innerWidth` and bare `innerWidth` resolve. Added in `lib/web/js/setup.ad`. `window.addEventListener('load'|'DOMContentLoaded', …)` / `window.onload` firing is now DONE in the DOM layer (round 9, `lib/web/dom/canvas.ad`: `window` gets the listener methods in `_js_build_document`, window listeners share the `DOC_EL` lifecycle, and `_fire_window_onload` runs `window.onload` after the load listeners). `test_jsengine_destructure_host.sh`.
- **R13** (`Object.hasOwn`/`Object.is` + RegExp `s`/`y` flags + signed-zero unary minus):
  - **`Object.hasOwn(obj, key)`** — the ES2022 static replacement for `hasOwnProperty` that dodges a shadowed/absent prototype method. Coerces the key to a property name and reports an OWN (not inherited) property, including array indices and the array `length`; a non-object first arg returns `false`. `object_has_own` in `lib/web/js/builtins/collections.ad`, registered in `setup.ad` (NAT_OBJECT_HASOWN).
  - **`Object.is(a, b)`** — the SameValue algorithm: identical to `===` except **NaN is equal to itself** and **+0 is distinct from -0** (via a `1/x` sign probe), with reference identity for objects and value equality for primitives/BigInt (falls through to `strict_eq`). `same_value` in `lib/web/js/builtins/collections.ad`, registered in `setup.ad` (NAT_OBJECT_IS).
  - **Unary minus preserves signed zero** — `eval_unary` (`lib/web/js/interp.ad`) now computes `U_NEG` as an IEEE negation (`-1.0 * x`) instead of `0.0 - x`, so `-(+0) === -0` (`1/-0 === -Infinity`). The old `0.0 - x` collapsed `-(+0)` back to `+0`, which broke `Object.is(0, -0)` and any `Math.sign`-of-negative-zero style logic. `String(-0)` stays `"0"` per spec.
  - **RegExp `s` (dotAll) flag** — `.` now also matches U+000A when the `s` flag is set (`re_da` global threaded from `obj_re_flags` bit3 through `regex_search` into the `RI_ANY` opcode); `/x/s.dotAll` is reflected, and dotAll flows through `exec`/`test`/`match`/`matchAll`/`replace`/`split`/`search` and the `new RegExp(src, "s")` constructor. `lib/web/js/{state,setup}.ad`.
  - **RegExp `y` (sticky) flag** — a sticky match must begin **exactly at `.lastIndex`** with no forward scan: `regex_search` returns `-1` after a single anchored attempt when sticky (bit4). `.sticky` is reflected; `exec`/`test` read+advance `.lastIndex` for `y` (not only `g`); and the `matchAll`/`replace`/`match` global loops honor the anchor so a gap stops iteration (correct tokenizer semantics). `lib/web/js/setup.ad`. `test_jsengine_es2022b_host.sh` (34 asserts vs node: hasOwn/is edge cases, `1/-0`, `/a.b/s`, sticky `.lastIndex` progression + non-contiguous stop + named-group×dotAll combo).
  - Regex flags: **`d` (indices) flag — now DONE (2026-07-18, gate `reindices`)**: `/re/d` reflects `hasIndices`, and a match result carries a parallel `.indices` array of `[start,end)` pairs per capture group (undefined for a non-participating group) plus `.indices.groups` (named-group pairs; undefined when there are none) — only present when `d` is set, threaded through both `exec()` and `matchAll()`. Exposure of the already-tracked `re_cap[]` start/end (`lib/web/js/setup.ad` `js_make_regex` bit5 + `regex_result_array`). STILL missing: **`u`/`v` (Unicode) mode** — the VM is byte-oriented, so `\u{...}` code-point classes and full surrogate handling remain out of scope.
- **R14** (HTML event-loop APIs — `requestAnimationFrame`/`cancelAnimationFrame` + the ordering guarantees frameworks depend on):
  - **Already present (verified, NOT rebuilt)** — `setTimeout`/`setInterval`/`clearTimeout`/`clearInterval`/`queueMicrotask` and the whole **microtask queue** landed with the Promise/async rounds (`lib/web/js/builtins/timers.ad` + `promise.ad`). Timers and promise reactions share ONE bounded deferred-callback table (`lib/web/js/state.ad`, `TIMER_CAP` 4096, `TIMER_DRAIN_CAP` 10000) that `js_eval` DRAINS at end-of-script in **(microtask-first, then delay, then insertion)** order — a `setTimeout(...,0)` scheduled before a `queueMicrotask`/`Promise.resolve().then` still runs AFTER it. NOT wall-clock async; the drain is deterministic and capped so a self-re-enqueuing `setInterval` cannot spin forever.
  - **`requestAnimationFrame(cb)` / `cancelAnimationFrame(id)`** (this round) — rAF callbacks register on the SAME timer table (`raf_schedule`, `timer_raf=1` flag) as one-shot **frame macrotasks**. Each enqueues into a frame indexed by `raf_sched_gen`; its ordering "delay" is the frame's **virtual timestamp** `(gen+1)*FRAME_MS_VC` (`FRAME_MS_VC=16.0`, ~60fps), which is also the `DOMHighResTimeStamp` handed to the callback at fire time. Because the timestamp is derived purely from a frame counter — **no `Date`, no random, no sleeping** — drains stay fully deterministic. Sharing the timer table means timers and frames live on ONE virtual timeline: a `setTimeout(...,0)` (delay 0) fires before frame 0 (t=16), and a microtask still beats both. A rAF that re-schedules a rAF from inside its own callback lands in the **NEXT** frame, not an infinite same-frame loop: `raf_sched_gen` is bumped the moment the first callback of the current frame fires (in `run_one_task`), so an animation loop advances 16→32→48… frame by frame and terminates at the drain cap. `cancelAnimationFrame` reuses `timer_clear` (shared id space). Added in `lib/web/js/builtins/timers.ad` (`raf_schedule` + `run_one_task` rAF branch), `state.ad` (`timer_raf`, `FRAME_MS_VC`, `raf_sched_gen`), `consts.ad` (NAT_REQUEST_ANIM_FRAME/NAT_CANCEL_ANIM_FRAME), `builtins/collections.ad` (dispatch), `setup.ad` (globals), reset in `api.ad` (`js_init`). `test_hambrowse_timers_host.sh` (15 asserts vs node/browser order: microtask-before-macrotask from BOTH `queueMicrotask` and `Promise.resolve().then`; timers by delay-then-insertion; clear cancels; interval-N-then-clear; args forwarded; nested timer→microtask; rAF runs with timestamp; cancelAnimationFrame; rAF-loop virtual-clock 16/32/48; and the shared timer+frame timeline).
  - Honest remaining gaps: rAF/timer ids share one namespace (spec keeps them separate — harmless in practice; real code doesn't cross-cancel); the model is eager end-of-script drain (no true suspension / no wall-clock scheduling / no real fps pacing — a `setTimeout(f, 5000)` runs immediately at drain, ordered by delay); no `setImmediate`/`process.nextTick`; `queueMicrotask` shares the macrotask table (flagged, not a separate FIFO structure) but drains with correct microtask-first priority.
- **R15** (self-contained Web-platform APIs — `URL` / `URLSearchParams` + `btoa` / `atob`):
  - **`new URL(url[, base])`** — a WHATWG-URL parser that splits into `href`/`protocol`/`username`/`password`/`host`/`hostname`/`port`/`origin`/`pathname`/`search`/`hash` own properties plus a live `searchParams`. The component splitter mirrors the one `lib/web/dom/canvas.ad` (`_url_parse_into`) already uses for `location`, EXTENDED here with userinfo (`user:pass@`) and a `searchParams` object. **Relative-to-base resolution** covers the cases real routers use: an already-absolute url (used as-is); scheme-relative `//host/x` (inherits base scheme); path-absolute `/x` (base origin + path); fragment-only `#h` and query-only `?q` (rebuild off the base minus its own fragment/query); and a relative segment `x` / `../x` / `./x` (replace the base's last path segment, then RFC-3986 `remove_dot_segments` fold — `new URL('../d','https://ex.com/a/b/c').pathname === '/a/d'`). A relative url with no absolute base throws **TypeError**. `toString()`/`toJSON()` return `href`. Added in `lib/web/js/builtins/url.ad` (`_url_split`/`_url_resolve`/`_url_remove_dots`/`_url_construct`), constructors + prototypes in `install_url_apis` (`setup.ad`). Honest limit: **`searchParams` is one-way** — it is populated from the initial query, but mutating it does NOT rewrite `url.search`/`url.href` (documented; avoids the accessor-backing machinery the object model lacks).
  - **`URLSearchParams(init)`** — from a `?a=1&b=2` query string OR a plain object of key→value pairs. Full method set: `get`/`getAll` (repeated keys preserved in order), `has`, `set` (overwrite the first match + drop the rest, or append when absent), `append`, `delete`, `toString`, `forEach(cb, thisArg)`, `entries`/`keys`/`values`, `sort` (stable by name), and a default `[Symbol.iterator]` === `entries()` so `for (const [k,v] of params)` works. Correct **application/x-www-form-urlencoded** codec: `toString` percent-encodes and maps space→`+`; parsing decodes `%XX` and `+`→space (`'a b'`→`'a+b'`, round-trips). Pairs are stored under hidden (`@@k`/`@@v`) parallel arrays so `Object.keys`/`JSON.stringify` never surface engine state.
  - **`btoa(str)` / `atob(b64)`** — standard RFC 4648 base64 with `=` padding. `btoa('Hello') === 'SGVsbG8='`; padding edges `btoa('M')==='TQ=='`, `btoa('Ma')==='TWE='`; `atob` tolerates ASCII whitespace (space/tab/newline/CR/FF), stops at `=`, and throws **InvalidCharacterError** on an invalid base64 char; full round-trip. Honest limit: this engine stores strings as BYTES (latin1 code units), so a code unit is always ≤ 0xFF — the spec's `btoa` >0xFF `InvalidCharacterError` cannot trigger for byte inputs (multi-byte UTF-8 is encoded byte-wise). Added in `lib/web/js/builtins/url.ad` (`_btoa`/`_atob`), globals via `def_global_native`.
  - Gate: `test_hambrowse_url_host.sh` (37 asserts — URL split + origin, all six relative-resolution shapes + dot-folding + no-base TypeError, URLSearchParams get/getAll/set/append/delete/toString encoding + repeated keys + iteration order + for-of + sort, base64 round-trip + padding + whitespace + validation). Zero regressions across the existing JS gates (`jsengine_host`, `es2022`, `es2022b`, `timers`, `structuredclone`, `typedarray`, `proxy`, `webstorage`).
- **R16** (REAL over-the-wire `fetch()` — the marquee capability separating a toy browser from a real one):
  - **What `fetch()` did before** — it was fixture-only: the engine (extern-free, dual-target) resolved `fetch(url)` synchronously against a JS-settable `__setFetchFixture` table, never touching the network. `lib/web/js/module.ad`'s comment noted "real http9 traffic is wired in a front-end", but nothing wired it — a page's `fetch()` could only see canned data.
  - **The kernel HTTP/DNS path used (NO sockets)** — Hamnix is Plan-9-shaped: networking is the `/net` FILE TREE, not `socket()`. The reusable client is `user/http9.ad`'s `http_get(url, dst, dst_cap, *out_status, *out_body_off, *out_body_len)`: it parses the URL, resolves the host via the in-kernel DNS resolver (`sys_resolve`, syscall 269 → `drivers/net/dns.ad`) or an IPv4 literal, connects through `user/net9.ad`'s `net_dial` / `net_dial_tls` (TLS for `https`), sends an `HTTP/1.0` GET, drains the response honoring `Content-Length` / chunked (decoded in place), and follows up to 6 redirects. This is the SAME client a page navigation and `<img>`/`<link>` use — proven on-device.
  - **The engine seam (stays extern-free)** — added `js_set_fetch_transport(fn)` (mirrors the existing `js_set_module_resolver` seam; `lib/web/js/api.ad` + `state.ad`). The engine holds only a raw address and, in `do_fetch` (`lib/web/js/builtins/fetch.ad`), calls the embedder's transport for an absolute `http(s)://` URL, then PARSES the returned raw HTTP bytes itself (`make_response_net`: status line → `status`/`statusText`, header block → a full **Headers** object, body slice → `__body`). A non-absolute URL (bare fixture key) still routes to the fixture table, so both coexist and every existing gate stays green. Transport returns NULL on a DNS/connect failure → the Promise **rejects** (TypeError); a non-2xx HTTP status **fulfills** with `ok=false` (Fetch-spec correct). `init.method`/`init.body` are forwarded.
  - **Richer Response/Headers** — Headers now stores EVERY parsed response header (ordered lowercased-name/value arrays on hidden slots) with case-insensitive **`get`**, new **`has`**, and **`entries()`** (`[[name,value],…]`); `text()`/`json()`/`arrayBuffer()` read the real body bytes. `NAT_HEADERS_HAS`/`NAT_HEADERS_ENTRIES` added (`consts.ad`, dispatch in `builtins/collections.ad`).
  - **On-device wiring (LANDED)** — `user/hambrowse.ad` registers an `http_get`-backed transport (`_js_fetch_transport`, into a dedicated `jsfetch_buf` kept separate from the page's `resp_buf`) via `js_set_fetch_transport` before the first `he_layout()`. ## FRESH AUDIT 2026-07-17 (verified against the modular lib/web/ tree)

The 2026-07-16 "High-impact worklist (56)" below was captured on the PRE-refactor monoliths
— its `lib/htmlengine.ad`/`lib/jsengine.ad` line numbers are dead, and many items have since
landed. A read-only re-audit against the current `lib/web/{js,html,css,dom,layout}/` tree +
the passing host gates found the following.

**RESPONSIVE + WEB-API ROUND 2026-07-18 (three genuinely-missing high-value W3C features, host-verified):**
- **[css-conditional] CSS `@media` queries — responsive CSS (NEW, gate `media`).** The engine had NO
  `@media` support: a `@media (...) { ... }` block was mis-parsed by the ruleset scanner (prelude read
  as a selector, inner rules as garbage decls) and silently dropped, so every responsive breakpoint was
  ignored. Now `lib/web/css/cascade.ad`'s sheet scanner (`_parse_sheet`) recognises `@`-at-rules
  (`_parse_at_rule`) and evaluates a **Media Queries Level 4 subset** against the LIVE viewport (`bw`×`bh`)
  via `_media_matches`/`_mq_query`/`_mq_feature`: media types (`screen`/`all` match; `print`/`speech` do
  not), `and`, comma (**OR** list), `not`, `only`, and the range/discrete features **`min-width`/
  `max-width`/`width`, `min-height`/`max-height`/`height`, `orientation` (landscape/portrait from bw vs bh),
  and `prefers-color-scheme`** (the engine renders LIGHT, so `dark` never matches). A matching block's inner
  rulesets cascade EXACTLY like top-level rules (same `_parse_ruleset` path, correct source-order
  specificity); a non-matching block is skipped; other block at-rules (`@keyframes`/`@font-face`/`@supports`/
  `@page`) are brace-matched and skipped, and statement at-rules (`@import`/`@charset`/`@namespace`) skip to
  `;`. Length tokens parse px/em/rem (em/rem ×16). Verified HOST-ONLY render-to-PNG (`test_hambrowse_media_host.sh`,
  fixture `hambrowse_media.html`, 16 pixel asserts): the SAME page rendered at widths **400/700/1000** flips
  each element's background per the breakpoint it should match (`max-width:500`, `min-width:800`,
  `min-width:600 and max-width:900` range, `orientation:landscape`), and a `prefers-color-scheme:dark` block
  never applies. **HONEST SCOPE / LIMITATIONS:** viewport width = `bw` (window content width — no separate
  device-width / scrollbar accounting); em/rem length units use a fixed 16px root (no cascade-resolved root
  font-size); truly-unknown features (`hover`/`pointer`/`resolution`/`aspect-ratio`/…) evaluate PERMISSIVELY
  (match) so a block gated on an unmodelled feature still contributes its rules (a deliberate don't-drop-content
  choice); the custom-property pre-sweep (`_collect_cvars`) still harvests `--vars` inside a non-matching
  `@media` block; and this is a STATIC evaluation at layout time (a resize re-lays-out and re-evaluates, but
  there is no live matchMedia/change-listener surface).
- **[ecmascript] RegExp `d` (indices) flag — DONE (gate `reindices`).** See the R13 regex-flags note below —
  `/re/d` now reflects `hasIndices` and exposes per-capture `[start,end)` `.indices` (+ `.indices.groups`)
  through `exec()`/`matchAll()`.
- **[html-forms] `FormData` — DONE (gate `formdata`).** The AJAX form multimap primitive; see live-worklist
  item #10 below. Reuses the URLSearchParams `@@k/@@v` storage + methods; only the constructor is distinct.

**CSS-ANIMATIONS + matchMedia ROUND 2026-07-18b (two genuinely-missing high-value W3C features, host-verified):**
- **[css] CSS animations — `@keyframes` + `animation`/`animation-name` STATIC END-STATE (NEW, gate `keyframes`).**
  The engine had NO animation support: `@keyframes` blocks were brace-skipped by `_parse_at_rule` and
  `animation`/`animation-name` declarations silently ignored, so an element that settles to a keyframed end
  colour (fade-in, colour-cycle, reveal) rendered its BASE (pre-animation) state. Now `lib/web/css/cascade.ad`
  records each `@keyframes NAME { … }` definition's SETTLED frame — the `100%`/`to` keyframe, else the last
  frame (`_parse_keyframes_body`) — as an INERT end-state pseudo-rule (`_store_keyframe`, tagged `r_kfname` so
  `_rule_matches` skips it in selector matching) keyed by the interned animation name (`_kf_intern`, a
  per-`_collect_css` name registry). A rule/element declaring `animation-name: X` or the `animation` shorthand
  (the name token found by skipping durations/timing-functions/keywords via `_anim_kw_skip`) resolves `X` to a
  winner (`m_animname`) and `_cascade_match_current` OVERLAYS that keyframe's **background / text-colour /
  transform / border-colour** onto the cascade winners, regardless of source order (keyframes may be defined
  after the rule that references them — both are parsed in one `_collect_css` pass, so the id lookup pairs them).
  Verified HOST-ONLY render-to-PNG (`test_hambrowse_keyframes_host.sh`, fixture `hambrowse_keyframes.html`):
  a `.fade` box with `animation:fadein …` settles to the `100%` frame `#00aa00` (over its base `#333`), a
  `.slide` box with `animation-name:reveal` settles to the `to` frame `#cc0000`, a no-animation box keeps its
  base `#112233`, and a box whose `animation` names a NON-EXISTENT keyframes keeps its base `#445566`.
  **HONEST SCOPE / LIMITATIONS:** this is the **settled END-STATE** only — the model renders the visually-final
  frame (equivalent to `animation-fill-mode: forwards` / the end of a finite animation), with **no timeline, no
  mid-frame interpolation, no live ticking** (a headless single render has no clock — consistent with the whole
  engine's static-render model). Overlay is limited to the four visually-dominant settle properties
  (background, colour, transform, border-colour); other keyframed properties (opacity, width/position offsets,
  per-side margins) are NOT yet overlaid. An `infinite` animation has no true final state — we render its end
  keyframe anyway (a reasonable static snapshot). `animation-fill-mode`/`-delay`/`-direction` are parsed-and-
  ignored beyond name resolution. **`transition` is likewise graceful:** a static render already shows the
  resting/final state a transition animates TO (there is no interactive state change to animate FROM), so
  `transition` needs no special handling — it is a no-op decl the parser skips, which is the correct rendered
  result.
- **[dom] `window.matchMedia(query)` — the JS surface of media queries (NEW, gate `matchmedia`).** SPAs branch
  on `window.matchMedia('(min-width:…)').matches` for responsive JS (lazy-loading, layout mode,
  prefers-color-scheme); the engine exposed `@media` in CSS but had NO `matchMedia`, so such a script threw
  (`matchMedia is not a function`). Now `lib/web/dom/canvas.ad` `_match_media` (NID_MATCHMEDIA, on `window` +
  the bare global like `getComputedStyle`) returns a **MediaQueryList** whose `.matches` is the LIVE evaluation
  of the query against the viewport via a new public `he_media_match(p,n)` in `lib/web/css/cascade.ad` — the
  SAME Media Queries L4 evaluator (`_media_matches`) the stylesheet's `@media` blocks use — plus `.media`
  (the query text), `.onchange` (null), and `addListener`/`removeListener`/`addEventListener`/
  `removeEventListener`/`dispatchEvent` (NID_MQL_NOOP). Verified HOST-ONLY (`test_hambrowse_matchmedia_host.sh`,
  fixture `hambrowse_matchmedia.html`, one page at TWO widths): `(min-width:600px)` is `true` at 880 / `false`
  at 400, `(max-width:500px)` flips the other way, the `(min-width:600) and (max-width:900)` range matches only
  at 880, `screen and (min-width:400)` matches, `print`/`prefers-color-scheme:dark` never match (screen/light
  render), `orientation:landscape` matches at 880×600 but not 400 (portrait), and the listener methods accept
  args without throwing. **HONEST SCOPE:** the listener registration is a **best-effort no-op** — the headless
  engine does ONE layout, so a `change` event never fires (there is no live resize/scheme-change source); the
  MediaQueryList's `.matches` is a correct point-in-time snapshot for the render's viewport.

**CSS `filter` ROUND 2026-07-18c (the #1 remaining CSS gap, host-verified):**
- **[css] CSS `filter` — `grayscale`/`brightness`/`invert`/`sepia`/`contrast`/`saturate`/`opacity`/`blur`
  (NEW, gate `filter`).** Filter Effects Module L1. Previously ABSENT (the round-2 audit's #1 CSS hole:
  hero-image treatments, glassmorphism, disabled/reader states). Followed the documented plumbing path exactly.
  `lib/web/css/cascade.ad` parses `filter:` in `_box_decl` via a new `_filter_value(vs,ve)` that packs the first
  recognised filter function into a single int `(type << 20) | amount` (amount = an integer percent —
  `grayscale(0.5)==grayscale(50%)==50` — for the colour filters, or a pixel radius for `blur`). It rides the
  cascade/rule machinery as a full parallel property (`d_filter`/`m_filter`/`r_filter[]`, reset + specificity
  tracked like `border-radius`) with **zero reflow cost** — the value flows through the untouched
  `layout/box.ad` record set. `box.ad` resolves the winner (inline `d_filter` over stylesheet `m_filter`) into
  a `box_filter_stack`, and stamps `bfill_filter[]` (parallel to `bfill_rgb[]`, set at ALL FOUR bfill emit
  sites — the block-box close, the inline-block chip, the blockquote bar, and the page-bg fill — so a stale
  index can never leak a filter). `he_bfill_filter(i)` (`dom/canvas.ad`) exposes it, and `htmlpage_render`
  (`lib/htmlpage.ad`) runs a per-pixel **POST-PASS** `htmlpaint_filter_rect(x,y,w,h,packed)` over each filtered
  box's rect AFTER all fills/glyphs/borders are painted — so the element AND its subtree content are filtered
  together (true CSS filter semantics, not just the background). `lib/htmlpaint.ad` implements the transforms on
  the internal framebuffer: grayscale/sepia/invert as Rec.601-luma / colour-matrix mixes, brightness/contrast as
  channel scales, saturate as luma-extrapolation, and `blur` as a **separable box blur** (two 1-D passes via a
  4096-wide scratch). Verified HOST-ONLY render-to-PNG (`test_hambrowse_filter_host.sh`, fixture
  `hambrowse_filter.html`, probe `scripts/hb_filter_probe.py`): eight solid boxes (one filter each) are
  pixel-scanned across the whole image — each box's ORIGINAL colour is GONE and the EXPECTED filtered colour is
  PRESENT (grayscale `#cc4422`→`#696969`, brightness `#4c4c4c`→`#727272`, invert `#204060`→`#dfbf9f`, sepia
  `#6688aa`→`#b09d7a`, contrast `#909090`→`#a0a0a0`, saturate `#a06040`→`#d15111`, opacity `#402010`→`#9f8f87`
  (approx), blur preserves the uniform interior + visibly smears the box's text), and an unfiltered control box
  is unchanged. Zero regressions (borderradius/boxshadow/website re-run green). **HONEST SCOPE / LIMITATIONS:**
  ONE filter function per element (a space-separated filter LIST collapses to the FIRST recognised function — no
  chaining); `drop-shadow()` is recognised (parsed, no crash) but painted as a **no-op** (its offset/blur/colour

  composite against in the static single-render model); `hue-rotate()` is not implemented; the effect covers the
  element's BORDER BOX (out-of-flow descendants painted outside the box are not filtered); `backdrop-filter`
  (filtering what is BEHIND the element) is a distinct feature, still absent. The filter runs only on the
  host render-to-PNG framebuffer path (a no-op when draws are redirected to an external RGBA target).

**TRUE PER-SIDE BORDERS ROUND 2026-07-18d (the #1 remaining CSS gap above, host-verified):**
- **[css-backgrounds-borders §4] true per-side border WIDTH + STYLE (NEW, gate `borderside`).** Previously a CSS
  border was a single UNIFORM 1px stroke — one width, one style — so a `border-bottom:2px solid` tab underline, a
  thick coloured accent side, or a dashed frame ALL collapsed to the same thin solid rectangle (the round-3
  audit's #1 CSS hole). Now `lib/web/css/cascade.ad` parses **`border-top`/`-right`/`-bottom`/`-left`** (and each
  `-width`/`-style`/`-color` longhand) plus the **`border-width`/`border-style` TRBL shorthands** (1..4 values,
  CSS top/right/bottom/left expansion) into two PACKED ints — `d_bordps` (8 bits/side width, `top | right<<8 |
  bottom<<16 | left<<24`, 0..120px) and `d_bordss` (4 bits/side style, `code+1` where 0 solid / 1 dashed / 2
  dotted / 3 double / 4 none). They ride the cascade/rule machinery as ONE parallel property pair (`r_bordps`/
  `r_bordss` + a shared `sbps` specificity slot → `m_bordps`/`m_bordss`, inline `d_` wins over sheet `m_` exactly
  like `d_bordc`), flow through the layout record set (`bbox_ps`/`bbox_ss` in `layout/box.ad`, emitted at block/
  float close alongside `bbox_rgb`), and `lib/htmlpage.ad` (`_hpg_paint_border_sides`/`_hpg_edge_h`/`_hpg_edge_v`)
  paints **FOUR INDEPENDENT EDGES** — each its own thickness and solid/dashed/dotted/double pattern (dashed/dotted
  = repeated `fill_rect` segments with an equal gap; double = two thin rules + a gap; none = skipped side).
  Cross-rule/decl composition works within a block: a uniform `border:2px solid` seeds all four sides so a
  following `border-bottom:5px dashed` overrides only the bottom (`_seed_sides_from_uniform`). Verified HOST-ONLY
  render-to-PNG (`test_hambrowse_borderside_host.sh`, fixture `hambrowse_borderside.html`, 18 pixel asserts via the
  gfx driver's arbitrary-pixel sampler): a box with a **12px left edge vs a 2px top edge** (ink 9px into the left,
  paper 2px past the top — distinct widths), a box with a **6px DASHED bottom vs a SOLID top** (bottom alternates
  ink/paper along its length, top stays continuous ink — dashed != solid), a uniform **`border:3px dashed`**
  showing an ink dash AND a paper gap on one edge, and a **`border-width:2px 10px 2px 10px`** TRBL shorthand
  giving a 10px left / 2px top. **ZERO REGRESSION proof:** a plain uniform `border:Npx solid` keeps `d_bordps==0`
  and stays on the legacy 1px `stroke_round_rect` path — **all 163 existing hambrowse fixtures render BYTE-IDENTICAL
  (ppm + display-list dump) between base and this change**; the pre-existing `border` gate stays green.
  **HONEST SCOPE / LIMITATIONS:** (1) per-side COLOURS are NOT yet modelled — the box carries ONE (uniform) border
  colour (`bbox_rgb`); the last `border-*-color` token wins for all four sides (the common single-accent-side
  pattern e.g. `border-bottom:2px solid #ccc` is correct because only one side is present, but a red-top/blue-bottom
  box paints one colour). (2) A uniform thick SOLID border (`border:5px solid`) still renders as the legacy 1px
  stroke (kept on the legacy path deliberately, for byte-identical regression); only a NON-solid uniform style
  (dashed/dotted/double) or an actual per-side longhand routes to the multi-px per-side painter. (3) Border widths
  are clamped to 0..120px (keeps the packed int positive, dodging the signed-`>>` quirk). (4) `border-radius` is
  honoured only on the legacy uniform stroke — a per-side-bordered box paints square corners (the four edges are
  axis-aligned fills; a rounded per-side border needs arc-segment stroking, deferred). (5) `groove`/`ridge`/
  `inset`/`outset` are accepted but rendered as `solid` (no 3-D bevel model). (6) Dashed/dotted dash lengths are a
  fixed heuristic (dash = 2×width, dot = 1×width, equal gaps), not the browser's exact dash algorithm.
