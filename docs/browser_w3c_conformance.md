# Native Browser (hambrowse) тАФ W3C / ECMAScript Conformance Roadmap

From a 14-agent parallel audit (2026-07-16). Dual-target, host-iterated (~40 gates). ~30-50% conformant/area; full conformance needs real spec ALGORITHMS per subsystem тАФ enabled by the modular lib/web/ tree refactor.

- **R8** (numeric literals + JSON reviver):
  - **Numeric literal forms** тАФ ES2021 numeric separators `_` in decimal/hex/binary/octal integers + fraction + exponent (`1_000_000`, `0xFF_FF`, `1e1_0`); binary `0b`/`0B` and octal `0o`/`0O` integer literals; BigInt `n` suffix (originally kept as a float64 Number тАФ **now a true arbitrary-precision BigInt as of R11**, see below). Fixed in `lib/web/js/lexer.ad` (lex_number) + `lib/web/js/value.ad` (parse_num_bytes). Previously the lexer split `1_000` into `1` + identifier `_000` (ReferenceError) and read `0b101` as `0` + identifier тАФ a hard abort of any modern minified/transpiled bundle using these forms. `test_jsengine_numliteral_host.sh`.
  - **JSON.parse(text, reviver)** тАФ the optional second argument is now honored: a bottom-up InternalizeJSONProperty walk calls `reviver.call(holder, key, value)` at every node (children first), deletes the property on an `undefined` result, handles arrays by index, threads the holder object as `this`, and invokes the root with key `""`. Previously the reviver was silently ignored (parsed data returned untransformed тАФ a correctness bug). Fixed in `lib/web/js/builtins/collections.ad` (json_parse_val + json_internalize/json_intern_props/json_reviver_call). `test_jsengine_numliteral_host.sh`. (JSON.stringify's `replacer` argument was subsequently landed in R10 тАФ see above.)
- **R9** (class private members + static blocks + global symbol registry):
  - **Class private `#fields` / `#methods` / accessors** тАФ the lexer now tokenizes `#name` as `TK_PRIVATE`, retaining the leading `#` so a private slot's key can never collide with a normal string property. Private instance fields (`#x = v`), private methods (`#m(){}`), private getters/setters (`get #x(){}`), and private static members (`static #n = v`) all parse and evaluate; `this.#x` / `Ctor.#n` access works through the ordinary member get/set path (stored under the `#`-prefixed key). The `#x in obj` **brand check** is a dedicated `ND_PRIVNAME` LHS handled by the `in` binop (never evaluated as a value; probes the private slot through the prototype chain). Crucially, private slots are HIDDEN: `Object.keys`, `for-in`, and `JSON.stringify` skip any key beginning with `#` (via a new `is_hidden_key` in `lib/web/js/value.ad`). Fixed across `lib/web/js/{lexer,parser,interp,value}.ad` + `builtins/collections.ad`. `test_jsengine_es2022priv_host.sh`.
  - **`Symbol.for(key)` / `Symbol.keyFor(sym)`** тАФ a process-wide symbol registry interns by string key so `Symbol.for("a") === Symbol.for("a")` (and stays distinct from a plain `Symbol("a")`); the registration key is stored back on the symbol as a hidden `@@forkey` own property for `Symbol.keyFor` reverse lookup (returns `undefined` for an unregistered symbol; throws `TypeError` on a non-symbol). Added in `lib/web/js/builtins/r7.ad` (symbol_for/symbol_key_for) + registered in `setup.ad`. `test_jsengine_es2022priv_host.sh`.
  - **`static { }` initializer blocks** тАФ the class-body parser now recognizes a bare `{ ... }` member after `static` (`ND_METHOD` with `n_c=3`); `eval_class` runs the block bodies interleaved with `static` fields in SOURCE ORDER, with `this` = the class object and the class's own name bound in a class-scope env (so `static { C.x = ... }`, `static y = C.z`, and `this.#priv` inside a block all resolve). Added in `lib/web/js/parser.ad` (parse_class) + `interp.ad` (eval_class static pass). `test_jsengine_es2022priv_host.sh`.
  - **Bonus fix (symbol-key leak)** тАФ the same `is_hidden_key` filter also stops symbol-keyed properties (`@@sym:` keys) from leaking into `Object.keys` / `JSON.stringify`, a pre-existing correctness bug (`o[sym]=v` previously showed `@@sym:NNN` in enumeration output).
- **R10** (`for await...of` + JSON.stringify replacer):
  - **`for await (x of asyncIterable)`** async iteration тАФ the parser recognizes a contextual `await` after `for` (inside an async body) and marks `ND_FOROF` (`n_d=1`); `exec_forof` dispatches to a new `exec_forawait` (`lib/web/js/interp.ad`). It first tries the source's `@@asyncIterator` (a fresh global `g_sym_async` symbol, also installed on the generator prototype so an `async function*` is its own async iterator): it calls `next()`, **awaits** the result via the engine's existing deterministic `do_await` (microtask-drain) to get `{value, done}`, and binds/executes the loop body per element (honoring `break`/`continue`/labels through the shared `forof_bind`/`forof_after`). If there is no `@@asyncIterator`, it falls back to the **async-from-sync wrapper**: it materializes the sync iterable via `collect_iterable_into` and **awaits each produced value** тАФ so `for await (x of [Promise.resolve(1), 2, ...])` yields the resolved values in order. Because `await` drains the deterministic queue, the whole loop completes synchronously (no call-stack suspension), matching the engine's eager async/generator model. `test_jsengine_asynciter_host.sh`.
  - **`JSON.stringify(value, replacer[, space])`** тАФ the 2nd argument is now honored. A **function replacer** `fn(key, value)` is called (with the holder as `this`) per property, its result substituted; an **array replacer** is an allowlist of object keys to include (applied at every object level, not to array indices). `value.toJSON(key)` is also honored on this path. Because a replacer function does its own string-building (which cannot run while the shared build cursor is held by `json_emit` тАФ the same two-phase hazard as `String.replace`), the value tree is **pre-transformed** into a fresh tree (`json_transform`/`json_rep_value`/`json_key_allowed`, root holder `{"": value}`) BEFORE the emitter runs; the no-replacer path stays on the fast direct emitter (byte-identical to before). Added in `lib/web/js/builtins/collections.ad` (`json_stringify_full` + helpers). `test_jsengine_asynciter_host.sh`.
- **R11** (true arbitrary-precision BigInt тАФ closes the largest ES gap):
  - **BigInt value type** тАФ a new `TAG_BIGINT` value tag (`lib/web/js/consts.ad`) whose `v_ref` indexes a new immutable bignum record: a **sign + base-2^32 little-endian limb magnitude** (`lib/web/js/bigint.ad`, pools `bi_sign`/`bi_off`/`bi_len` over a shared `bi_limbs` uint32 bump pool). Records are immutable, so a sign flip shares limbs with no copy. `123n` / `0xffn` / `0b1010n` / `0o17n` literals now LEX to a distinct `TK_BIGINT` token carrying the source digits (lexer keeps the text sans `n`; the regex/divide heuristic treats a BigInt literal as a value so `7n / 2n` is division, not a regex) and PARSE to `ND_BIGINT`, evaluated by exact `bi_parse` тАФ no more float64 precision loss past 2^53.
  - **Arithmetic** тАФ exact `+ - * / % **` on two BigInts: schoolbook magnitude add/sub/mul, **bit-by-bit long division** (├╖ **truncates toward zero**, `%` takes the dividend's sign), and exponentiation-by-squaring. `++`/`--` and compound `+= -= *= тАж` stay BigInt. Mixing a BigInt with a Number in an arithmetic/bitwise op throws **TypeError** (per spec); unary `+` on a BigInt throws; `/ 0n` and a negative `**` exponent throw **RangeError**.
  - **Comparisons** тАФ `< > <= >=` and `== ===` including exact **BigInt-vs-Number numeric compare** (`2n < 3`, `5n == 5` true, `5n === 5` false тАФ `===` is false across types); loose `==` also handles BigInt-vs-String/Boolean by mathematical value.
  - **Bitwise** тАФ `& | ^` implemented with **infinite-width two's-complement** semantics (correct for negative operands via a to/from two's-complement conversion); `<<` is a magnitude shift keeping sign; `>>` is an **arithmetic (floor) shift** (`-8n >> 1n === -4n`, `-7n >> 1n === -4n`); `~x === -(x+1)`. `>>>` throws TypeError (BigInts have no unsigned shift).
  - **Conversions / methods** тАФ `typeof 1n === "bigint"`; `String(1n)` / `` `${1n}` `` / `1n.toString(radix)` (2тАУ36); truthiness (`0n` falsy); `BigInt(number|string|boolean|bigint)` (non-integer Number тЖТ RangeError); `BigInt.asIntN(bits,x)` / `BigInt.asUintN(bits,x)`. `JSON.stringify` on a BigInt (or an object containing one) throws **TypeError**, per spec. Console/display prints the digits with no trailing `n` (String() semantics). Touched: `lib/web/js/{bigint,consts,state,lexer,parser,interp,setup}.ad` + `builtins/{collections,strings}.ad`. `test_jsengine_bigint_host.sh` (40 asserts vs known results: `2n**100n`, `50!`, div/mod signs, two's-complement bitwise, mixing/JSON TypeErrors).
  - Coverage is honest and broad: bitwise AND/OR/XOR/shifts (incl. negatives), `asIntN`/`asUintN`, and all six arithmetic ops are DONE. NOT done: `BigInt64Array`/`BigUint64Array` typed arrays, `DataView` get/setBigInt64, and a `Number(bigint)` that round-trips huge magnitudes exactly (uses a float64 approximation for out-of-int64 comparison fallbacks only).
  - Next JS gap: async iterator `return()`/`throw()` propagation and `Symbol.asyncIterator` on user objects returning promises from `next()` (partially exercised; async generators covered); then `BigInt64Array`/`DataView` BigInt accessors.
- **R12** (destructuring ASSIGNMENT + `window`-shaped globals):
  - **Destructuring as an ASSIGNMENT statement** тАФ `[a,b]=[b,a]` (swap), `[x,,y]=arr` (elision/skip), `[a,...rest]=arr` (rest), `({p,q}=obj)`, `({name:nm}=obj)` (rename), object rest `({a,...others}=obj)`, element/property **defaults** (`[m=5]`, `{v=5}`), nested patterns (`[p,[q,r]]=тАж`, `({user:{id,tags:[t0,t1]}}=тАж)`), and member/index leaf targets (`[o.x,a[0]]=тАж`) all work. Previously this did NOTHING: `assign_to` handled only ident/member/index LHS, so an array/object **literal** LHS was silently ignored тАФ only the `let/const/var` DECLARATION path (`bind_pattern`) understood patterns. Root cause + fix: added `assign_array_pattern`/`assign_object_pattern` in `lib/web/js/interp.ad`, dispatched from `assign_to` on an `ND_ARRAY`/`ND_OBJECT` target; they walk the literal's element list and assign each leaf via `assign_to` (recursing for nested patterns). Because `eval_assign` already fully evaluates the RHS before calling `assign_to`, the swap is correct. Two supporting parser fixes (`lib/web/js/parser.ad`): array-literal **elisions are now preserved** as a new `ND_HOLE` node (`parse_array_lit`) so `[x,,y]=arr` maps `yтЖРarr[2]` (a hole also now reads as `undefined` with correct `.length` in a value context, via `eval_array_lit`); and object shorthand **CoverInitializedName** `{x = default}` parses (wrapping the shorthand in an `ND_ASSIGN`) so `({v=5}=obj)` applies its default.   - **`window`-shaped globals** тАФ the existing `window`/`globalThis`/`self` alias now also carries the browser viewport/URL data properties pages read ungated: `innerWidth`/`innerHeight`/`outerWidth`/`outerHeight` (default 1024├Ч768), `scrollX`/`scrollY`/`pageXOffset`/`pageYOffset` (0), `devicePixelRatio`/`screenX`/`screenY`, a minimal `location` object (`href`/`protocol`/`host`/`hostname`/`port`/`pathname`/`search`/`hash`/`origin`), and a `screen` object. Defined on the global scope (like `navigator`) so both `window.innerWidth` and bare `innerWidth` resolve. Added in `lib/web/js/setup.ad`. `window.addEventListener('load'|'DOMContentLoaded', тАж)` / `window.onload` firing is now DONE in the DOM layer (round 9, `lib/web/dom/canvas.ad`: `window` gets the listener methods in `_js_build_document`, window listeners share the `DOC_EL` lifecycle, and `_fire_window_onload` runs `window.onload` after the load listeners). `test_jsengine_destructure_host.sh`.
- **R13** (`Object.hasOwn`/`Object.is` + RegExp `s`/`y` flags + signed-zero unary minus):
  - **`Object.hasOwn(obj, key)`** тАФ the ES2022 static replacement for `hasOwnProperty` that dodges a shadowed/absent prototype method. Coerces the key to a property name and reports an OWN (not inherited) property, including array indices and the array `length`; a non-object first arg returns `false`. `object_has_own` in `lib/web/js/builtins/collections.ad`, registered in `setup.ad` (NAT_OBJECT_HASOWN).
  - **`Object.is(a, b)`** тАФ the SameValue algorithm: identical to `===` except **NaN is equal to itself** and **+0 is distinct from -0** (via a `1/x` sign probe), with reference identity for objects and value equality for primitives/BigInt (falls through to `strict_eq`). `same_value` in `lib/web/js/builtins/collections.ad`, registered in `setup.ad` (NAT_OBJECT_IS).
  - **Unary minus preserves signed zero** тАФ `eval_unary` (`lib/web/js/interp.ad`) now computes `U_NEG` as an IEEE negation (`-1.0 * x`) instead of `0.0 - x`, so `-(+0) === -0` (`1/-0 === -Infinity`). The old `0.0 - x` collapsed `-(+0)` back to `+0`, which broke `Object.is(0, -0)` and any `Math.sign`-of-negative-zero style logic. `String(-0)` stays `"0"` per spec.
  - **RegExp `s` (dotAll) flag** тАФ `.` now also matches U+000A when the `s` flag is set (`re_da` global threaded from `obj_re_flags` bit3 through `regex_search` into the `RI_ANY` opcode); `/x/s.dotAll` is reflected, and dotAll flows through `exec`/`test`/`match`/`matchAll`/`replace`/`split`/`search` and the `new RegExp(src, "s")` constructor. `lib/web/js/{state,setup}.ad`.
  - **RegExp `y` (sticky) flag** тАФ a sticky match must begin **exactly at `.lastIndex`** with no forward scan: `regex_search` returns `-1` after a single anchored attempt when sticky (bit4). `.sticky` is reflected; `exec`/`test` read+advance `.lastIndex` for `y` (not only `g`); and the `matchAll`/`replace`/`match` global loops honor the anchor so a gap stops iteration (correct tokenizer semantics). `lib/web/js/setup.ad`. `test_jsengine_es2022b_host.sh` (34 asserts vs node: hasOwn/is edge cases, `1/-0`, `/a.b/s`, sticky `.lastIndex` progression + non-contiguous stop + named-group├ЧdotAll combo).
  - Regex flags STILL missing (documented, not yet built): **`d` (indices) flag** тАФ needs per-match capture-index arrays exposed as a `.indices` property on the result (the VM tracks `re_cap[]` start/end already, so this is exposure work, deferred to keep the diff focused); **`u`/`v` (Unicode) mode** тАФ the VM is byte-oriented, so `\u{...}` code-point classes and full surrogate handling remain out of scope.
- **R14** (HTML event-loop APIs тАФ `requestAnimationFrame`/`cancelAnimationFrame` + the ordering guarantees frameworks depend on):
  - **Already present (verified, NOT rebuilt)** тАФ `setTimeout`/`setInterval`/`clearTimeout`/`clearInterval`/`queueMicrotask` and the whole **microtask queue** landed with the Promise/async rounds (`lib/web/js/builtins/timers.ad` + `promise.ad`). Timers and promise reactions share ONE bounded deferred-callback table (`lib/web/js/state.ad`, `TIMER_CAP` 4096, `TIMER_DRAIN_CAP` 10000) that `js_eval` DRAINS at end-of-script in **(microtask-first, then delay, then insertion)** order тАФ a `setTimeout(...,0)` scheduled before a `queueMicrotask`/`Promise.resolve().then` still runs AFTER it. NOT wall-clock async; the drain is deterministic and capped so a self-re-enqueuing `setInterval` cannot spin forever.
  - **`requestAnimationFrame(cb)` / `cancelAnimationFrame(id)`** (this round) тАФ rAF callbacks register on the SAME timer table (`raf_schedule`, `timer_raf=1` flag) as one-shot **frame macrotasks**. Each enqueues into a frame indexed by `raf_sched_gen`; its ordering "delay" is the frame's **virtual timestamp** `(gen+1)*FRAME_MS_VC` (`FRAME_MS_VC=16.0`, ~60fps), which is also the `DOMHighResTimeStamp` handed to the callback at fire time. Because the timestamp is derived purely from a frame counter тАФ **no `Date`, no random, no sleeping** тАФ drains stay fully deterministic. Sharing the timer table means timers and frames live on ONE virtual timeline: a `setTimeout(...,0)` (delay 0) fires before frame 0 (t=16), and a microtask still beats both. A rAF that re-schedules a rAF from inside its own callback lands in the **NEXT** frame, not an infinite same-frame loop: `raf_sched_gen` is bumped the moment the first callback of the current frame fires (in `run_one_task`), so an animation loop advances 16тЖТ32тЖТ48тАж frame by frame and terminates at the drain cap. `cancelAnimationFrame` reuses `timer_clear` (shared id space). Added in `lib/web/js/builtins/timers.ad` (`raf_schedule` + `run_one_task` rAF branch), `state.ad` (`timer_raf`, `FRAME_MS_VC`, `raf_sched_gen`), `consts.ad` (NAT_REQUEST_ANIM_FRAME/NAT_CANCEL_ANIM_FRAME), `builtins/collections.ad` (dispatch), `setup.ad` (globals), reset in `api.ad` (`js_init`). `test_hambrowse_timers_host.sh` (15 asserts vs node/browser order: microtask-before-macrotask from BOTH `queueMicrotask` and `Promise.resolve().then`; timers by delay-then-insertion; clear cancels; interval-N-then-clear; args forwarded; nested timerтЖТmicrotask; rAF runs with timestamp; cancelAnimationFrame; rAF-loop virtual-clock 16/32/48; and the shared timer+frame timeline).
  - Honest remaining gaps: rAF/timer ids share one namespace (spec keeps them separate тАФ harmless in practice; real code doesn't cross-cancel); the model is eager end-of-script drain (no true suspension / no wall-clock scheduling / no real fps pacing тАФ a `setTimeout(f, 5000)` runs immediately at drain, ordered by delay); no `setImmediate`/`process.nextTick`; `queueMicrotask` shares the macrotask table (flagged, not a separate FIFO structure) but drains with correct microtask-first priority.
- **R15** (self-contained Web-platform APIs тАФ `URL` / `URLSearchParams` + `btoa` / `atob`):

Added in `lib/web/js/builtins/url.ad` (`_url_split`/`_url_resolve`/`_url_remove_dots`/`_url_construct`), constructors + prototypes in `install_url_apis` (`setup.ad`). Honest limit: **`searchParams` is one-way** тАФ it is populated from the initial query, but mutating it does NOT rewrite `url.search`/`url.href` (documented; avoids the accessor-backing machinery the object model lacks).

Correct **application/x-www-form-urlencoded** codec: `toString` percent-encodes and maps spaceтЖТ`+`; parsing decodes `%XX` and `+`тЖТspace (`'a b'`тЖТ`'a+b'`, round-trips). - **R16** (REAL over-the-wire `fetch()` тАФ the marquee capability separating a toy browser from a real one):
  - **What `fetch()` did before** тАФ it was fixture-only: the engine (extern-free, dual-target) resolved `fetch(url)` synchronously against a JS-settable `__setFetchFixture` table, never touching the network. `lib/web/js/module.ad`'s comment noted "real http9 traffic is wired in a front-end", but nothing wired it тАФ a page's `fetch()` could only see canned data.
  - **The kernel HTTP/DNS path used (NO sockets)** тАФ Hamnix is Plan-9-shaped: networking is the `/net` FILE TREE, not `socket()`. The reusable client is `user/http9.ad`'s `http_get(url, dst, dst_cap, *out_status, *out_body_off, *out_body_len)`: it parses the URL, resolves the host via the in-kernel DNS resolver (`sys_resolve`, syscall 269 тЖТ `drivers/net/dns.ad`) or an IPv4 literal, connects through `user/net9.ad`'s `net_dial` / `net_dial_tls` (TLS for `https`), sends an `HTTP/1.0` GET, drains the response honoring `Content-Length` / chunked (decoded in place), and follows up to 6 redirects. This is the SAME client a page navigation and `<img>`/`<link>` use тАФ proven on-device.
  - **The engine seam (stays extern-free)** тАФ added `js_set_fetch_transport(fn)` (mirrors the existing `js_set_module_resolver` seam; `lib/web/js/api.ad` + `state.ad`). The engine holds only a raw address and, in `do_fetch` (`lib/web/js/builtins/fetch.ad`), calls the embedder's transport for an absolute `http(s)://` URL, then PARSES the returned raw HTTP bytes itself (`make_response_net`: status line тЖТ `status`/`statusText`, header block тЖТ a full **Headers** object, body slice тЖТ `__body`).
