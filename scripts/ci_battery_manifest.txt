# 'hamnix-bare-metal' round-robin-shards this list across a matrix of

# boots serially blew past the 40-min budget (~75 min needed) and the
# whole battery went permanently red, providing ZERO signal. Add a gate:
# drop a line here (order = fast→slow keeps shards balanced under round-
# robin). Wrapped gates (ci_run_gate.sh) emit the three-valued verdict
# (INCONCLUSIVE→warning on a TCG-starved runner); direct ones are hard
# PASS/FAIL. Blank lines and # comments are ignored.

# FIXED + RE-GATED — test_ext4_largedir's "Multiply-claimed block(s) ... 2109
# shared between /lgdir and an unrelated inode" was NOT the htree growth: the
# growth legitimately allocated a genuinely-free block. The real bug was
# LEAKED INODES. ext4_free_inode was group-0-only (it REFUSED any inode past
# group 0), the asymmetric twin of the already-multi-group ext4_alloc_inode —
# so deleting a high-group inode never cleared its bitmap bit. Worse, unlink/
# rmdir/rename-overwrite freed inodes WITHOUT honouring i_links_count or
# marking them deleted (i_links_count=0 + i_dtime), so a freed inode kept
# looking IN-USE to e2fsck while its data block was reclaimed and reallocated
# to /lgdir's growth -> multiply-claim. A boot mkdir/rmdir smoke test left an
# unconnected directory inode (links=2, extent->2109) behind exactly this way.
# Fix: ext4_free_inode is now multi-group, and every delete path routes through
# _ext4_drop_inode_link (decrement link if >1; else trim blocks + mark deleted
# before releasing), so e2fsck reports no multiply-claimed block / unconnected
# directory (only the intentional inode-2 dirent artifact + allocator drift).

# --- FIXED + RE-GATED: the cold NCQ block-path deadlock (GHC.IE) -------------
# test_ahci_blk, test_ahci_recovery and test_blk_sched used to deterministically
# WEDGE the guest the first time they drove a cold blk_read/blk_write (generic
# ahci_read/write_sectors -> _ncq_block_rw_sync, the SLEEPING NCQ FPDMA path)
# against the active AHCI port AFTER sched_init. Root cause: the driver set
# GHC.AE but NEVER GHC.IE — the GLOBAL AHCI interrupt-enable gate (spec §3.1.2).
# With GHC.IE clear the HBA suppresses every port's PxIE-armed source and never
# asserts its INTx line, so ahci_irq_handler never ran (ahci.irq_count stayed 0)
# and the sleeper on wq_wait_commit_timeout had NO waker; the jiffies-deadline
# backstop could not save it either because the LAPIC tick barely fires in the
# IF=0-heavy native boot (jiffies frozen ~7 while wall time was ~1.3s). Warm
# survived because the blkpath selftest busy-polls _ncq_irq_complete_port to
# drain, needing neither IRQ nor timeout; cold submitted and slept forever.
# Fixed by setting GHC.IE right after the IRQ handler is registered + the IOAPIC
# pin routed (drivers/ata/ahci.ad, commit "fix(ahci): set GHC.IE ..."). Verified
# under BOTH KVM and TCG: the completion IRQ fires and wakes via a real data-wake
# (rc=0), not the timeout and not a poll downgrade — the NCQ FPDMA fast path is
# genuinely exercised. All three now run to a deterministic PASS; re-gated below.
bash scripts/ci_run_gate.sh scripts/test_ahci_blk.sh
bash scripts/ci_run_gate.sh scripts/test_blk_sched.sh
