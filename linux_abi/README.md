# linux_abi/

Generated Linux 6.12 ABI definitions consumed by Hamnix.

Hamnix aims for binary-level compatibility with stock Linux kernel
modules. Rather than transcribing struct layouts by hand (and slowly
drifting), this directory holds machine-extracted layouts from a real
Linux 6.12 `vmlinux` image's BTF (BPF Type Format) section.

## Layout

```
linux_abi/
  README.md           — this file
  TARGET_ABI.md       — pinned Linux version + rationale
  structs/
    list_head.ad      — autogenerated from BTF
    kref.ad
    kobject.ad
    module.ad
```

Every file under `structs/` is autogenerated by
`scripts/gen_linux_abi.py`. The first few comment lines record the
source struct name, the header file it came from, the Linux version,
and the BTF type ID. **Do not hand-edit these files.**

## Regenerating

On a system that exposes BTF via `/sys/kernel/btf/vmlinux` (any
recent Debian/Ubuntu kernel with `CONFIG_DEBUG_INFO_BTF=y`):

```
python3 scripts/gen_linux_abi.py
```

Or point at an explicit raw-BTF blob:

```
python3 scripts/gen_linux_abi.py path/to/vmlinux.btf
```

If no BTF is available, the script prints instructions for fetching
one (e.g. `apt install linux-image-6.12-generic` or building from
source + `pahole --btf_encode_detached`) and emits **mock** struct
files marked with a `MOCK:` warning in the header. Mock files exist
solely so L1 work can compile against a known shape; their offsets
are hand-translated estimates and MUST be re-verified against real
BTF before any correctness claim.

## Validation in CI

```
python3 scripts/gen_linux_abi.py --check
```

This re-parses the BTF blob and asserts every `structs/*.ad` file
matches what BTF reports. Exit status:

| code | meaning                                       |
|------|-----------------------------------------------|
| 0    | all files match, or generation succeeded      |
| 1    | no BTF available (mock files emitted, or `--check` aborted) |
| 2    | `--check` found a mismatch                    |

Run this whenever bumping the pinned kernel version in
`TARGET_ABI.md` — it's the canonical way to catch a layout change.

## Adder type spellings

The generator maps BTF kinds to Adder types as follows:

| BTF kind             | Adder type                  | notes                       |
|----------------------|-----------------------------|-----------------------------|
| INT (n bits)         | `int<n>` / `uint<n>`        | signed flag → `int` prefix  |
| PTR                  | `uint64`                    | 8 bytes on x86_64           |
| ARRAY of T, N        | `T[N]`                      | recursively rendered        |
| STRUCT/UNION         | `opaque_bytes[size]`        | nested struct kept opaque   |
| ENUM / ENUM64        | `int32` / `int64`           | sized to BTF report         |
| FUNC_PROTO           | `uint64`                    | function pointer            |
| bitfield member      | `opaque_bits`               | L0 doesn't crack bitfields  |

Nested structs are kept opaque at this layer — callers that need to
reach into e.g. `module.list` cast it to a `list_head` view at the
matching offset. This keeps each `.ad` file freestanding (no
cross-file imports) and matches how the existing
`kernel/list.ad` was hand-written.

## Scope

L0 deliberately covers only four structs (`list_head`, `kref`,
`kobject`, `module`). Adding more is a one-line change to the
`TARGET_STRUCTS` tuple in `scripts/gen_linux_abi.py`.
